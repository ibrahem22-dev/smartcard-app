# SmartCard — Key Lifecycle & Auth Binding Contract

**Spec ID:** SEC-CONTRACT-001 · **Version:** 1.0 · **Status:** Draft for review
**Date:** 2026-06-21 · **Owner:** Security Agent
**Binds:** STORE-01 (MMKV encryption) · AUTH-01 (biometric + PIN)
**Platform:** React Native + Expo (dev client / prebuild — `react-native-mmkv` needs native code) · iOS + Android

> This is the **implementation contract**. STORE-01 and AUTH-01 must satisfy every `MUST`
> clause below to pass merge review. Each clause has an ID so reviews can cite pass/fail
> against it directly. `MUST` = blocking. `SHOULD` = required unless explicitly waived in PR.

---

## 0. Threat model (the one paragraph this whole spec defends against)

**Attacker:** has physical possession of the device (lost/stolen/seized), and may have root/jailbreak
and a copy of the app binary. **Does not** have the user's live biometric or PIN.
**Goal:** read balance / income / last-4 / card metadata at rest.
**Defense invariant:** *Without a fresh biometric or PIN success this session, the only thing on disk
is ciphertext, and the key that decrypts it is not retrievable.* Every requirement below exists to keep
that one sentence true.

---

## 1. Key lifecycle (the chain)

```
                           FIRST LAUNCH (once per install)
  ┌────────────────────────────────────────────────────────────────────────┐
  │ KGEN  generate 256-bit Data Encryption Key (DEK) via CSPRNG             │
  │ KWRAP store DEK in hardware-backed keychain, flagged auth-required      │
  │       iOS    : Keychain item, Secure Enclave-backed where available     │
  │       Android: Keystore key, StrongBox where available                  │
  │ KVER  write keySchemeVersion = 1 alongside                              │
  └────────────────────────────────────────────────────────────────────────┘
                                     │
                         EVERY SESSION (on unlock)
  ┌────────────────────────────────────────────────────────────────────────┐
  │ AUTH   biometric  OR  PIN→Argon2id  →  success unwraps keychain access  │
  │ KGET   retrieve DEK from keychain (OS prompts for biometric/passcode)   │
  │ MMKV   MMKV.initialize({ encryptionKey: DEK })                          │
  │ MEM    hold DEK in memory ONLY for the session                          │
  └────────────────────────────────────────────────────────────────────────┘
                                     │
                    BACKGROUND > 5 min  OR  explicit lock
  ┌────────────────────────────────────────────────────────────────────────┐
  │ ZERO   wipe DEK from memory · close MMKV instance · gate navigator      │
  │        next foreground → return to AUTH                                 │
  └────────────────────────────────────────────────────────────────────────┘
```

The DEK is **never** written anywhere except the hardware keychain. It is never in MMKV,
never in AsyncStorage/SecureStore-as-plain, never in Redux/Zustand state, never logged.

## 2. Key generation — `KGEN`

| ID | Requirement |
|----|-------------|
| KGEN-1 | DEK **MUST** be 256-bit, generated by a CSPRNG (`expo-crypto getRandomBytesAsync(32)` or platform secure RNG). |
| KGEN-2 | DEK **MUST NOT** be derived from device ID, install ID, timestamp, PIN, or any guessable/low-entropy value. |
| KGEN-3 | DEK **MUST** be generated exactly once per install (on first launch) and reused thereafter — not regenerated per session. |
| KGEN-4 | Source code **MUST NOT** contain any hardcoded key, seed, or fallback key (M9). |

---

## 3. Key storage — `KWRAP` (hardware keychain)

| ID | Requirement |
|----|-------------|
| KWRAP-1 | DEK **MUST** be stored in the platform hardware-backed keychain — iOS Keychain (Secure Enclave where available), Android Keystore (StrongBox where available). |
| KWRAP-2 | The keychain item **MUST** be flagged auth-required so the OS gates retrieval: iOS `accessControl: biometryCurrentSet` (falls back to device passcode); Android `setUserAuthenticationRequired(true)`. |
| KWRAP-3 | The keychain item **MUST** use a non-exportable, this-device-only accessibility class: iOS `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`; Android key non-exportable by construction. **No iCloud/Android backup sync of the DEK.** |
| KWRAP-4 | `expo-secure-store` is acceptable **only** with KWRAP-2 flags verified present on *both* platforms; it is **not** assumed hardware-backed otherwise. Default (unflagged) SecureStore is **MUST NOT** for the DEK. |
| KWRAP-5 | `keySchemeVersion` **MUST** be persisted next to the DEK to permit future migration without bricking installed data. |

---

## 4. Auth-gated retrieval & memory-only session — `KGET` / `MEM` / `ZERO`

| ID | Requirement |
|----|-------------|
| KGET-1 | DEK retrieval **MUST** occur only after an auth success this session, and the OS biometric/passcode prompt (triggered by KWRAP-2) **MUST** be the gate — not an app-level boolean. |
| KGET-2 | MMKV **MUST** be initialized with the retrieved DEK; an MMKV instance **MUST NOT** be opened before auth. |
| MEM-1 | The DEK **MUST** live in memory only. **MUST NOT** be persisted to MMKV, AsyncStorage, SecureStore, app state, or any log/breadcrumb. |
| MEM-2 | The biometric *result* (the "did auth pass" flag) **MUST NOT** persist across sessions (your checklist D). |
| ZERO-1 | On background > 5 min OR explicit lock OR app termination, the DEK reference **MUST** be cleared from memory and the MMKV instance closed. Clearing the *key* — not just flipping a flag — is what enforces the timeout. |

## 5. Auth ↔ key binding (state machine)

```
        cold start / DEK not in memory
                   │
                   ▼
            ┌────────────┐   biometric OR PIN success    ┌──────────────┐
            │  LOCKED    │ ─────────────────────────────▶│  UNLOCKED    │
            │ ciphertext │                                │ DEK in memory│
            │ only; no   │◀───────────────────────────── │ MMKV open;   │
            │ navigator  │   bg>5min / lock / fail-lockout│ nav mounted  │
            └────────────┘                                └──────────────┘
                   ▲                                              │
                   └──────────  ZERO wipes DEK  ──────────────────┘
```

| ID | Requirement |
|----|-------------|
| BIND-1 | Auth success **MUST** be the trigger that unwraps the DEK. No path may reach `UNLOCKED` (DEK in memory, navigator mounted) without it. |
| BIND-2 | Financial screens **MUST** mount only inside the authenticated navigator (ref STR-H01). Per-screen `useEffect` auth checks are **MUST NOT** as the sole gate. |
| BIND-3 | Cold start via deep link or OS state-restoration **MUST** route to `LOCKED` and force auth before the authenticated navigator mounts. |
| BIND-4 | There **MUST** be no app state in which the navigator is mounted but the DEK is absent (and vice versa) — they transition together. |

## 6. PIN fallback hardening — Argon2id + lockout

The PIN is the weakest link by entropy (4–6 digits). These clauses prevent it from being a brute-force backdoor.

| ID | Requirement |
|----|-------------|
| PIN-1 | The PIN **MUST NOT** be stored in any form that can be compared in plaintext. Store only a verifier, or use the PIN solely to unwrap key access via a slow KDF. |
| PIN-2 | KDF **MUST** be **Argon2id** (scrypt acceptable as a documented fallback). **MUST NOT** be MD5/SHA1/plain-SHA256 (M5). |
| PIN-3 | Argon2id params **MUST** be tuned to ~250–500 ms on a mid-range target device. Starting point: `m = 19456 KiB (19 MiB)`, `t = 2`, `p = 1`, 16-byte random salt, 32-byte output. Record the chosen params + `kdfVersion`. |
| PIN-4 | Salt **MUST** be a per-install 16-byte CSPRNG value, stored in the keychain (not bundled, not hardcoded). |
| PIN-5 | **Lockout MUST** apply with escalating backoff (see schedule). The biometric path's failures and the PIN path's failures share one counter — an attacker **MUST NOT** be able to reset the count by switching methods. |
| PIN-6 | A terminal action **MUST** be defined and implemented: after the final threshold, either wipe local financial data (DEK destroyed → ciphertext unrecoverable) or enter extended lock. (Product decides which; reviewer requires *one* be chosen.) |
| PIN-7 | The PIN **MUST NOT** unlock anything the biometric can't — no elevated/"backdoor" capability. It is a fallback of equal scope, hardened to compensate for lower entropy. |

**Lockout schedule (default — adjust with rationale):**

| Cumulative failures | Consequence |
|---------------------|-------------|
| 1–4 | allowed, no delay |
| 5 | 30 s lock |
| 6 | 1 min |
| 7 | 5 min |
| 8 | 15 min |
| 9 | 1 hour |
| 10 | **terminal action** (PIN-6) |

> Where the OS keychain enforces its own attempt limits (KWRAP-2), prefer/stack that enforcement —
> it resists app-process tampering that an app-level counter cannot.

## 7. Session timeout — `ZERO` triggers

| ID | Requirement |
|----|-------------|
| SESS-1 | An AppState listener **MUST** record background-entry time; on foreground, if elapsed > 5 min, run `ZERO` and require re-auth (M4). |
| SESS-2 | The 5-min check **MUST** use a monotonic/elapsed-time source resistant to wall-clock manipulation where the platform allows. |
| SESS-3 | On backgrounding, financial screens **MUST** be obscured (privacy/blur overlay) so the OS app-switcher snapshot never captures balance/income (ref AUTH-M01). |
| SESS-4 | Biometric enrollment change (new finger/face) **SHOULD** invalidate cached access and force re-auth — `biometryCurrentSet` provides this; confirm enabled. |

## 8. Security module interface contract

All of the above lives behind one auditable boundary: `/src/security/`. Nothing outside it touches the keychain or the DEK.

```ts
// /src/security/keyVault.ts  — sole owner of the DEK
interface KeyVault {
  isInitialized(): Promise<boolean>;          // does a DEK exist for this install?
  initializeOnFirstLaunch(): Promise<void>;   // KGEN + KWRAP + KVER (idempotent)
  unlockWithBiometric(): Promise<UnlockResult>;
  unlockWithPin(pin: string): Promise<UnlockResult>; // PIN-2..PIN-6 enforced here
  // Returns a handle, NOT the raw key. Raw DEK never leaves this module.
  getEncryptedStorage(): EncryptedStorageHandle; // throws if LOCKED
  lock(): void;                               // ZERO: wipe DEK from memory, close MMKV
}

type UnlockResult =
  | { ok: true }
  | { ok: false; reason: 'bad_credential' | 'locked_out'; retryAfterMs?: number };
```

| ID | Requirement |
|----|-------------|
| API-1 | The raw DEK **MUST NOT** be returned across the `/src/security/` boundary. Callers receive a storage handle, never the key bytes. |
| API-2 | `engines/` **MUST NOT** import from `security/` or `stores/` — engines stay pure (ref STR-L01). |
| API-3 | Network calls **MUST NOT** exist in `security/` — it is offline by construction. |

## 9. Acceptance criteria — review gate for STORE-01 / AUTH-01

A PR for STORE-01 or AUTH-01 **passes review only if** all are demonstrable:

- [ ] **AC-1** With the app locked, on-disk MMKV file inspected via debugger/USB is ciphertext; no plaintext balance/income/last-4 recoverable. *(invariant from §0)*
- [ ] **AC-2** Keychain item carries the auth-required + this-device-only flags on **both** iOS and Android. *(KWRAP-2, KWRAP-3)*
- [ ] **AC-3** No DEK, key bytes, salt, or PIN appears in any `console.*`, Sentry breadcrumb, or persisted store. *(MEM-1, MEM-2, logging checklist C)*
- [ ] **AC-4** Deep link / cold-start into a financial route lands on `LOCKED` and forces auth. *(BIND-3)*
- [ ] **AC-5** Switching biometric↔PIN does not reset the failure counter; lockout schedule fires; terminal action works. *(PIN-5, PIN-6)*
- [ ] **AC-6** PIN verification path uses Argon2id with recorded params; grep finds no MD5/SHA1/plain-SHA256 on the PIN. *(PIN-2, PIN-3)*
- [ ] **AC-7** Background > 5 min wipes the in-memory DEK (verified: post-timeout read fails until re-auth) and app-switcher snapshot is blurred. *(ZERO-1, SESS-1, SESS-3)*
- [ ] **AC-8** No hardcoded key/seed in source; `keySchemeVersion` + `kdfVersion` present. *(KGEN-4, KWRAP-5, PIN-3)*
